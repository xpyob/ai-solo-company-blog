# 从 4 个 Bot 到五岗协作：一人 AI 公司的第一期架构实践

> Status: draft  
> Version: GitHub professional edition  
> Scope: phase1 methodology and implementation review

## 背景与动机

很多关于 AI Agent 的分享，停留在两个方向：一是工具清单，二是愿景叙事。前者解决不了组织问题，后者落不到执行现场。

这次一期变革的目标很具体：验证一个人能否把多个 Agent 从“功能助手集合”升级成“可运行的一人公司系统”。

变革前，系统由 `main`、`smart`、`tools`、`imagine` 四个 bot 组成，已经具备基础分工，但仍存在几个典型问题：

- 组织口径不清，角色边界依赖临场理解
- 任务委派后容易失联，缺少统一回传和升级机制
- 能力认知模糊，知道“能做很多”，但无法系统盘点
- 方法论停留在脑内，没有沉淀成可复用执行资产
- 记忆系统不稳定，高价值信息无法稳定召回
- 根文件职责混杂，`MEMORY.md` 一度承担了过多角色

因此，核心问题不是“Agent 数量不够”，而是缺少一套结构化公司操作系统。

一期变革最后落下来的答案，是一个五层结构：

1. 组织层
2. 协作协议层
3. 能力层
4. ITTO 执行层
5. 知识层

它不是概念图，而是一套已经被真实任务验证过的实现方案。

---

## 架构设计：五层结构为什么必要

### 1. 组织层：先定义谁负责什么

多 Agent 系统的第一步不是多开几个实例，而是定义岗位。

一期完成后，组织从 4 岗升级为 5 岗：

- `main`：总控、收口、验收、优先级管理
- `smart`：分析、判断、方案推演
- `tools`：工程执行、工具链、系统落地
- `imagine`：表达、包装、传播内容
- `research`：研究、检索、情报沉淀

这里的关键不是“会什么”，而是“负责什么”。角色一旦按 owner 设计，协作成本会明显下降。

### 2. 协作协议层：解决多 Agent 最常见的失控问题

组织层只解决“谁来做”，不解决“怎么协作不失联”。

这一层是一期开工后才被明确补上的。实践证明，多 Agent 最容易失败的地方不是能力不够，而是：

- 委派后无回音
- 多人重复做同一件事
- 长结果只存在聊天里
- 卡点没人升级
- 下游完成后没有标准回传

因此一期新增了四份长期制度文件：

- `team-directory-protocol.md`
- `shared-file-protocol.md`
- `escalation-protocol.md`
- `handoff-template.md`

这一步相当于给系统加上“通信协议”和“交付协议”。

### 3. 能力层：把“感觉能做”变成能力地图

Agent 团队如果没有能力层，会一直停留在经验驱动状态。

一期做了三件事：

- 定义 10 大能力模块
- 锁定 16 项正式 skills 清单
- 对 skills 做 A/B/C 三层评估与映射

结果很清楚：当前强项集中在 AI 基础设施、信息管理、知识资产；短板集中在 CRM、会议闭环、安全质量。

这让后续补强不再靠感觉，而是可以按模块推进。

### 4. ITTO 执行层：把方法论写成执行资产

很多系统失败，不是因为没有想法，而是方法没有落到文件和流程。

一期将执行体系正式化为 9 份 ITTO 文件，覆盖：

- 组织协作
- 会议执行闭环
- CRM
- 安全与质量保障
- 经营驾驶舱
- 品牌与内容工业化
- 信息优先级管理
- AI 基础设施
- 基础设施自动化

这样做的价值很直接：流程不再依赖某一轮对话，而是变成可复用、可交接、可审查的系统资产。

### 5. 知识层：让系统具备长期累计能力

如果没有知识层，多 Agent 协作会不断重复过去的问题。

一期把知识与记忆分成不同职责：

- `MEMORY.md`：长期入口，保持极瘦
- `memory/`：角色记忆、规则记忆、群组记忆
- `knowledge/`：方法论、案例、结构化知识
- `shared/`：共享草稿、项目材料、可交接交付物

这套设计的核心思想是：入口轻，细节下沉，长结果文件化。

## 关键实现细节

## 一、组织实现：根文件拆职，不再混写

一期首先重写了根文件体系，核心不是“补文档”，而是拆职责。

典型结构如下：

```text
workspace-xxx/
├── AGENTS.md
├── IDENTITY.md
├── SOUL.md
├── TOOLS.md
├── MEMORY.md
├── memory/
└── knowledge/
```

关键规则：

- `AGENTS.md` 定义操作契约和协作规则
- `IDENTITY.md` 定义岗位职责与边界
- `SOUL.md` 定义表达气质与默认工作倾向
- `TOOLS.md` 记录本地环境说明
- `MEMORY.md` 只保留长期入口，不做流水账

这个拆分直接解决了根文件职能重叠的问题，也为后续岗位适配提供了模板。

## 二、协作实现：用文件协议替代聊天堆积

为了避免聊天承载过多长内容，一期明确了共享文件优先原则。

例如：

```text
shared/
├── projects/
├── research/
├── tables/
├── drafts/
├── archive/
└── articles/
```

实际规则包括：

- 长报告、多版本稿件、研究材料优先入共享文件
- 聊天里只回传摘要、状态、路径、遗留问题
- 每个任务必须有唯一 owner
- 下游完成后必须按模板回传

标准回传格式被约束为：

```text
完成状态
核心结果
文件路径
遗留问题
建议下一步
```

这比“做好了你看看”更适合团队运行。

## 三、能力实现：从清单到分层，而不是无限加工具

一期并没有走“哪里弱就继续装 skill”的路线，而是先做能力诊断。

其中一个重要结论是：能力缺口不一定要靠新工具补。有些短板更适合由协作协议和 ITTO 弥补。

例如：

- 会议闭环问题，本质更偏流程设计
- 安全质量问题，本质更偏检查机制
- CRM 问题，本质更偏经营动作标准化

这个判断避免了系统陷入“持续装插件，但组织能力不增长”的误区。

## 四、记忆实现：先保证可用，再追求本地最优

`memory_search` 在早期长期不稳，最后定位到 qmd reranker 的 context size 不够。

一期采取的方案不是等待上游修复，而是先做工程级止血：

- 将 `workspace-main/memory` 的 47 个文件、123 个 chunks 纳入索引
- 将 `searchMode` 改为 `vsearch`
- 启用 remote fallback：`OpenAI text-embedding-3-small`
- 保留本地索引，等待未来无缝切回

这个决策体现的是一人公司非常重要的原则：稳定可用优先于局部优雅。

## 五、协作验证：必须通过真实任务回归测试

架构只有在真实链路里跑通，才算有效。

一期做了实际验证：

```text
main 派任务
→ research 独立检索
→ research 按协议回传
→ main 收口并更新状态
```

验证结论很明确：

- 委派链路可用
- 回传格式可用
- 跨 Agent 协作不是人工复制粘贴
- 协作协议层在真实场景下有效

这一步比写更多说明文档更重要，因为它证明系统已经从“设计态”进入“运行态”。

## 踩坑与经验

### 1. 最大误判：以为组织 + 能力就够了

一期前半段最关键的修正，是补出“协作协议层”。

实践证明，组织层负责定义角色，能力层负责定义能做什么，但真正决定系统能否稳定运作的，是协作机制。

### 2. 记忆系统不要一开始就追求完美

本地方案不稳时，先上 fallback，先保证召回正确率。记忆系统是底盘，不该被“理想架构”拖死。

### 3. 根文件一定要瘦身

如果 `MEMORY.md`、`AGENTS.md`、`SOUL.md` 混写，系统很快会重新退化成大杂烩。文件职责清晰，是后续扩展的前提。

### 4. 多 Agent 不是多人设，而是多 owner

角色差异不能只停留在语气和人格层。真正有效的多 Agent，必须落实到责任边界、默认输入、交付格式和回传机制。

### 5. 能力建设不能只靠加工具

技能数量增长，并不等于公司能力增长。能力地图、协议层和执行层，往往比新增一个 skill 更重要。

## 数据总结

一期变革的关键数据如下：

- 岗位数：`4 → 5`
- 根文件：`25` 份（5 个 workspace × 5 份）
- 长期制度文件：`8` 份
- 角色记忆：`6` 份
- 群组记忆：`7` 份
- ITTO 文件：`9` 份
- 正式 skills：`16` 项
- 里程碑：`9` 个，全部完成
- memory 索引规模：`47` 文件 / `123` chunks
- Telegram 群组：`6` 个（1 个总控群 + 5 个双人群）

如果只用一句话总结这次实践，可以概括为：

> 一人 AI 公司的关键，不是先拥有多少 Agent，而是先把组织、协作、执行和知识沉淀成系统。

从结果看，一期变革完成的不是“多了一个 bot”，而是把一个松散的 Agent 组合，重构成了一个可继续扩展、可验证、可交接的一人公司底座。

后续如果继续扩张，重点已经不是再讲概念，而是继续沿这五层结构补齐运营、经营和质量控制能力。
